# Project Prompt: Control-M Job Trigger Automation Using Python and Configuration Files

## Project Objective

I want to automate the manual process of triggering Control-M jobs that internally trigger IICS workflows.

Currently, I manually login to a Linux server using PuTTY, execute multiple `pmrun` commands one by one, and provide my password whenever prompted.

The goal is to build a Python-based automation utility where:

* I provide a list of Control-M short job names and execution dates through an input file.
* I provide my password securely at runtime.
* Python automatically generates and executes required `pmrun` commands.
* All Control-M jobs should be triggered in parallel from a single execution.
* Execution details should be logged for tracking and troubleshooting.

---

# Current Manual Process

## Step 1: Login to Server

I manually login using PuTTY.

Authentication details:

* AD username
* Hostname
* Server password

There are three environments:

1. DEV
2. PRE-PRD
3. PRD

Each environment has:

* Different server.
* Different trigger script path.
* Separate deployment location for Python files and configuration files.

---

## Step 2: Execute Control-M Trigger Command

After logging into the required server, I manually execute commands like:

```bash
pmrun -u /path/iics_wf_trigger.sh ABCD123 2026-07-11
```

Command explanation:

```
pmrun
    |
    |-- Control-M command used to trigger a job

-u /path/iics_wf_trigger.sh
    |
    |-- IICS workflow trigger shell script

ABCD123
    |
    |-- Control-M short job name

2026-07-11
    |
    |-- Execution/business date
```

Important:

`ABCD123` is a **Control-M short job name**, not an IICS workflow name.

Python does not need to know the actual IICS workflow details. It only needs to trigger Control-M jobs.

---

## Step 3: Password Authentication

After executing each `pmrun` command, the system prompts for my password.

Requirements:

* Password validity is one day.
* I should provide the password during runtime.
* Password should not be stored permanently.
* Password should not be written into logs.
* Password should only exist in memory during execution.

Example:

```
Enter Password:
********
```

PuTTY authentication remains manual.

Python starts execution only after the user is already logged into the Linux server.

# Control-M Job Input File Requirement

Daily job execution details should come from a separate CSV file.

File:

```
jobs.csv
```

Example:

```csv
controlm_job_name,execution_date
ABCD123,2026-07-11
XYZ001,2026-07-10
CUSTOMER_LOAD_JOB,2026-07-09
ACCOUNT_LOAD_JOB,2026-07-01
```

Explanation:

* `controlm_job_name` = Control-M short job name passed to pmrun.
* `execution_date` = Date argument passed to pmrun.

Requirements:

* Support any number of jobs.
* Support different dates for each job.
* User should only update this file daily.
* No Python code changes should be required.

# Parallel Execution Requirement

This is the most important requirement.

If there are 40 Control-M jobs:

Example:

```
JOB001
JOB002
JOB003
...
JOB040
```

All jobs should be triggered simultaneously.

Do NOT execute sequentially.

Incorrect:

```
JOB001
(wait)

JOB002
(wait)

JOB003
```

Required:

```
JOB001 ─┐
JOB002 ─┤
JOB003 ─┤
JOB004 ─┤
JOB005 ─┘

All triggered together
```

Requirements:

* No hardcoded parallel limit.
* No maximum job restriction.
* Assume server has capacity to execute multiple jobs.
* Use Python concurrency mechanisms.
