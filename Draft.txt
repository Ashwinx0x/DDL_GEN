# Project Prompt: IICS Workflow Trigger Automation Using Python and Configuration Files

## Project Objective

I want to automate the manual process of triggering multiple IICS workflows through Control-M server commands.

Currently, I manually login to a Linux server using PuTTY, execute multiple `pmrun` commands one by one, and provide my password whenever prompted.

The goal is to build a Python-based automation utility where:

* I provide a list of workflows and execution dates through an input file.
* I provide my password securely at runtime.
* Python automatically generates and executes all required `pmrun` commands.
* All workflows should trigger in parallel from a single execution.
* Execution details should be logged for tracking and troubleshooting.

---

# Current Manual Process

## Step 1

I login to a server using PuTTY.

Authentication details:

* AD username
* Hostname
* Server password

There are three environments:

1. DEV
2. PRE-PRD
3. PRD

Each environment has a different server and different paths.

---

## Step 2

After login, I manually execute commands like:

Example:

```bash
pmrun -u /path/iics_wf_trigger.sh ABCD123 2026-07-11
```

Where:

* `/path/iics_wf_trigger.sh` = IICS trigger shell script path
* `ABCD123` = workflow/job name
* `2026-07-11` = workflow execution date

The actual command format should be configurable.

---

## Step 3

After executing the command, the system prompts me for my IICS password.

The password:

* Is valid for one day.
* Should be provided during runtime.
* Should not be stored anywhere permanently.

# Automation Requirements

## 1. Python should NOT automate PuTTY login

Important:

PuTTY login will remain manual.

The Python script will execute only after I am already logged into the required Linux server.

Flow:

```
Login through PuTTY manually

        ↓

Run Python script

        ↓

Enter IICS password

        ↓

Python triggers workflows
```

# 2. Configuration File Requirement

Create a configuration file to store environment-specific details.

Use YAML format.

Example:

`config.yaml`

```yaml
environment: DEV

trigger_script_path: /apps/dev/iics_wf_trigger.sh

command_template: "pmrun -u {script_path} {workflow_name} {execution_date}"
```

For different environments, only configuration changes.

Example:

DEV:

```yaml
environment: DEV
trigger_script_path: /apps/dev/iics_wf_trigger.sh
```

PRE-PRD:

```yaml
environment: PREPRD
trigger_script_path: /apps/preprd/iics_wf_trigger.sh
```

PRD:

```yaml
environment: PRD
trigger_script_path: /apps/prd/iics_wf_trigger.sh
```

The Python code should remain the same across environments.

Requirements:

* Each workflow can have a different execution date.
* User should only modify this file daily.
* Python code should not require changes.

The Python script must request the password at runtime.

Example:


Use secure input.

Requirements:

* Password should not be displayed while typing.
* Password should not be stored.

Use Python secure password input methods.

# 5. Parallel Execution Requirement

This is the most important requirement.

If there are 40 workflows:

Example:

```
JOB001
JOB002
JOB003
...
JOB040
```

All should start together.

Do NOT execute sequentially.

Avoid:

```
JOB001
(wait)
JOB002
(wait)
JOB003
```

Required:

```
JOB001  ─┐
JOB002  ─┤
JOB003  ─┤
JOB004  ─┤── Execute simultaneously
JOB005  ─┘
```

There should be:

* No hardcoded parallel limit.
* No restriction like maximum 5 or 10 jobs.
* Assume server can handle multiple workflows.

Use Python concurrency libraries.
